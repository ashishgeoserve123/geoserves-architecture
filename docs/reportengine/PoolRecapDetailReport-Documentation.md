## 1. What Is This Report?

The Pool Recap Detail Report is one sheet inside the **Pool Performance Recap Report** Excel workbook. It gives a passage-by-passage, report-by-report breakdown of how a vessel performed against its pool terms during a given half-year period.

The report answers:
- How many nautical miles and hours did each passage take?
- How much fuel (ME, AUX, OTHERS) was consumed per report?
- Was a report exempt from pool performance evaluation — and why?
- What is the vessel's speed and consumption at pool speed, per passage, in good weather?

The report is generated by the local .NET service (`ReportService`) and embedded as a worksheet tab inside the combined Recap workbook.

---

## 2. Entry Point — How the Report Starts

The Recap report flow runs in 5 steps via the API:

```
POST /Report/NewReportRequest/PoolPerformanceRecapReport   → creates a report job, returns reportId
GET  /Scheduler/FetchSectionsForReport/{reportId}          → returns list of sectionIds to process
POST /Scheduler/ProcessSection/{sectionId}                 → runs the data fetch + computation for that section
GET  /Scheduler/ReportDocGeneration/{reportId}             → generates the Excel file
GET  /ScorpioReport/DownloadReportFile/{reportId}          → returns the .xlsx file
```

`ReportService.cs → GenerateDocument()` is the main orchestrator for the Excel generation step.

It calls `PoolRecapReportV2.WriteDetailSheet()` to generate and attach the Detail sheet.

`WriteDetailSheet()` calls `PoolRecapDetailRepository.GetDetailRows()` to fetch all data, then passes the result to `PoolRecapDetailReport.GenerateDocument()` to write the Excel sheet.

## 3. Step 1 — SQL Query (What Gets Fetched and How)

**Code:** `PoolRecapDetailRepository.Queries.cs → BuildMainQuerySql(whereClause)`

The main SQL query is a single CTE-based SELECT that returns one row per voyage report for the vessel in the requested date/passage range.

### CTE 1 — `VesselPoolConfig_Raw`

This CTE finds the pool configuration for the vessel.

It joins three tables:
- `Vessels` — the vessel record (contains `ImoNumber`, `PoolTermsId`)
- `PoolTermsAssignedToVessel` — links vessels to specific pool term assignments
- `PoolTerms` — the actual pool term record with speeds, thresholds, and factor settings

**Deduplication:** Multiple active pool term records may exist for the same vessel (e.g. two rows in `Vessels` share the same IMO). A `ROW_NUMBER()` window function partitions by `ImoNumber` and keeps only one row. Priority is: rows where `PoolTermsId` is directly set on the vessel come first; then by most recent `SFPM_PoolTermsAssignedToVesselId`.

The CTE produces one row containing:
- `PoolTermBallastSpeed`, `PoolTermLadenSpeed` — the evaluation speeds (knots) from pool terms
- `MinPassageHours` — minimum hours a passage must have to not be "Passage too Short"
- `WindSpeedThreshold`, `WaveHeightThreshold` — weather exclusion limits
- `BadDayHours`, `BadDayHoursId` — bad weather hour threshold and its unit type
- `Multiplier` — power factor used in ME/Day at Pool Speed formula (default 2.5)
- `CurrentFactorId`, `WindSpeedUnitId`, `WaveHeightUnitId` — unit system flags used for unit conversion in C#
- `VesselEvalSpeedBallast`, `VesselEvalSpeedLaden` — optional per-vessel override speeds (used instead of pool term speeds if non-zero)

### CTE 2 — `VesselPoolConfig`

Filters `VesselPoolConfig_Raw` down to `rn = 1` — keeps only the top-priority pool config row per vessel.

### CTE 3 — `AnalyzedWeather_Agg`

Pre-aggregates the `AnalyzedWeather` table to one row per `FormId`:
- `AVG` of wind and wave → used as a fallback 24h weather value
- `MAX` of the `IsWindExceed` and `IsWaveExceed` flags — if any hourly record in that form had a flag, the aggregated row inherits it

### Final SELECT

Joins `Forms` (the voyage reports table) to `VesselPoolConfig` and `AnalyzedWeather_Agg`.

Returns 35+ columns per report including:
- `Location` — port name (from/to)
- `Type` — report type: "Departure (ATD)", "Noon", "Arrival (ATA)"
- `LoadCondition` — "Laden" or "Ballast"
- `ReportDateTimeUTC`, `ReportDateTime` (local)
- `SteamingHrs` — hours from DB (will be overridden in Step 3)
- `ObsDistance` — observed nautical miles since previous report
- `OrderedSpeed` — the CP speed the vessel was ordered to maintain
- `ObsSpeed` — speed over ground
- `RPM`, `Slip`, `ShaftPower`, `EffectiveCurrent`
- `IsWindExceed`, `IsWaveExceed`, `IsPoolExclude` — pre-computed exclusion flags
- Pool config columns: `PoolTermBallastSpeed`, `PoolTermLadenSpeed`, `MinPassageHours`, `Multiplier`, etc.
- Fuel columns (`ME_HSFO`, `ME_VLSFO`, etc.) are all `NULL` in SQL — populated in C# in Step 7

**Ordering:** Results are ordered `DESC` by `VoyageNo` then `ReportDateTimeUTC` — newest passage first, within each passage newest report first. The C# layer reverses this during passage grouping to process chronologically.

## 4. Step 2 — Row Mapping

**Code:** `PoolRecapDetailRepository.Mapper.cs → MapToDetailRows(table)`

The `DataTable` returned from SQL is mapped column-by-column into a `List<PoolRecapDetailRow>`. Each property uses typed helpers (`Get<T>`, `GetNullable<T>`, `GetStr`, `ParseDecimal`) to safely read values and handle nulls.

One special case: `ReportDateTime` (local time). The mapper uses `GetLocalDateTime()` which reads the stored local time string and parses it as an **unspecified-kind** `DateTime` — this preserves the original local time without the C# runtime adding a machine-timezone offset.

## 5. Step 3 — Steam Hours Recalculation

**Code:** `PoolRecapDetailRepository.cs → RecalculateSteamHours(rows)`

The `SteamingHrs` value stored in the `Forms` DB table can be incorrect (for example, a form might report 4 hours when the actual elapsed time was 14 hours).

To fix this, the system recomputes `SteamHours` for every row by measuring the actual time gap between consecutive UTC report times:

```
SteamHours[i] = (ReportDateTimeUTC[i] - ReportDateTimeUTC[i-1]).TotalHours
```

Rules:
- Rows are processed in order of `ReportDateTimeUTC` ascending.
- ATD (Departure) rows are **skipped** — they are the starting reference point and have no previous row to diff against.
- The computation resets at each ATD — the next row after an ATD is measured from that ATD's UTC time.
- The result replaces `row.SteamHours` in-place.

This recalculation runs **before** event deductions, so the event deductions subtract from the corrected hours.

## 6. Step 4 — Pool Term Mappings (Fuel Bucket Config)

**Code:** `PoolRecapDetailRepository.Allocations.cs → GetPoolTermMappings(poolTermId)`

This query fetches the fuel classification rules for the pool term. It joins four tables:

- `PoolTermsConsumptionCategoryMapping` — links a pool term to consumption category mappings
- `ConsumptionCategory` (parent) — the top-level category (ME, AUX, OTHERS)
- `ConsumptionCategory` (child) — the specific allocation name (e.g. "Main Engine", "Auxiliary Boiler", "Incinerator")
- `PoolTermConsumptionCategoryFuelGroupings` — links category mappings to specific fuel types
- `FuelTypes` — the fuel type record (HSFO, VLSFO, LSMGO, etc.)

Each returned row (`PoolTermCategoryMapping`) contains:
- `ParentName` — the section: "ME", "AUX", or "OTHERS"
- `ChildName` — the specific allocation label that will match raw allocation names from the ROB system
- `PoolFuelBucketId` — 1 = HSFO, 2 = VLSFO, 3 = LSMGO (which fuel bucket this maps to)
- `IsBioFuel` — whether this is a bio-fuel type (matched by `FuelClassId`)
- `FuelTypeId` — the specific fuel type ID for exact matching

These mappings are the "lookup table" used in Step 7 to classify raw allocation amounts.

## 7. Step 5 — Raw Allocations

**Code:** `PoolRecapDetailRepository.Allocations.cs → GetRawAllocations(formIds)`

This query fetches the actual fuel consumption amounts recorded per report.

It joins:
- `Robs` → `Rob` → `Allocation`

Where `EventRobsRowId IS NULL` — this filter is critical. It means we only fetch **main form-level allocations**, not allocations tied to event ROB rows. Event allocations are handled separately and then subtracted (Step 6).

Each returned row (`RawAllocation`) contains:
- `FormId` — which voyage report this allocation belongs to
- `FuelTypeId` — which fuel type
- `FuelType` — the label of the fuel (informational only, not used for classification)
- `FuelClassId` — 1 = bio-fuel, otherwise 0
- `AllocationName` — the consumption category label (e.g. "Main Engine", "Incinerator") — this is what gets matched against `ChildName` in the pool term mappings
- `Amount` — the quantity consumed in metric tonnes

## 8. Step 6 — Event Detection and Deductions

Some voyage reports contain **EventROBsRows** — special ROB rows logged during specific events within a reporting period (e.g. slow-down orders, deviations, canal transits). These events have their own separate steam hours, distance, and fuel consumption. They need to be **subtracted** from the main form totals so the pool evaluation only sees the "normal" portion of each report.

This happens in three phases:

### Phase A — Find Which Forms Have Events

**Code:** `Events.cs → GetFormsWithEvents(formIds)`

A simple SQL `EXISTS` check on `EventROBsRow` for each `FormId`. Returns a `HashSet<long>` — the set of form IDs that have at least one event row.

### Phase B — Subtract Event Fuel Allocations

**Code:** `Events.cs → GetEventAllocations(eventFormIds)` + `SubtractEventAllocations(rawAllocations, eventAllocations)`

`GetEventAllocations` fetches the same `Robs → EventROBsRow → Rob → Allocation` join but for the event ROB rows only (the inverse filter: `EventRobsRowId IS NOT NULL`).

`SubtractEventAllocations` then mutates the main `rawAllocations` list in-place:
1. Build a lookup keyed by `(FormId | FuelTypeId | FuelClassId | AllocationName)`.
2. For each event allocation matching a key in the main list, subtract the event amount.
3. Clamp the result to 0 — never go negative.
4. Remove any entry whose amount has been reduced to exactly 0.

After this step, the raw allocations represent only the non-event portion of each report's fuel consumption.

### Phase C — Subtract Event Steam Hours and Distance

**Code:** `Events.cs → GetEventDeductions(eventFormIds)` + `ApplyEventDeductions(rows, deductions)`

`GetEventDeductions` queries `EventROBsRow` directly, aggregating per `FormId`:
- `EventSteamHours` = `SUM(DATEDIFF(MINUTE, EventROBStartDateUTC, EventROBEndDateUTC) / 60.0)`
- `EventDistance` = `SUM(EventROBObsDistance)`

`ApplyEventDeductions` then subtracts these from every non-ATD row:
```
row.SteamHours = max(0, row.SteamHours - deduction.SteamHours)
row.ObsDistance = max(0, row.ObsDistance - deduction.Distance)
```

## 9. Step 7 — Fuel Allocation Classification

**Code:** `FuelAllocationClassifier.cs → Classify(mappings, allocations)`

This is the core classification step. It takes:
- `mappings` — the pool term fuel rules (from Step 4)
- `allocations` — the event-adjusted raw amounts (from Steps 5 + 6)

And produces a `Dictionary<long, FuelAllocationResult>` — one `FuelAllocationResult` per `FormId` containing classified and summed fuel totals.

### How Classification Works

For each raw allocation:

**Step A — Resolve Section (which top-level category?)**

`ResolveSection(mappings, allocationName)`:
- Normalise the `AllocationName` (strip all whitespace, lowercase).
- Scan all mappings and normalise their `ChildName` the same way.
- If a match is found → return the mapping's `ParentName` ("ME", "AUX", or "OTHERS").
- If no match → return "OTHERS" as fallback.

**Step B — Resolve Bucket (which fuel type?)**

`ResolveBucket(mappings, allocationName, fuelTypeId, fuelClassId)`:
- Again normalise and scan for a `ChildName` match.
- From the matching mappings, find the right bucket in this priority order:
  1. If `FuelTypeId` matches exactly → use that bucket immediately.
  2. If `IsBioFuel == true` and `fuelClassId == 1` → use that bucket.
- Returns `PoolFuelBucketId` (1, 2, or 3) or 0 if no match.

**Step C — Guard for bucket = 0**

If the resolved bucket is 0 (no match), check the section:
- If section is **ME** or **AUX** → skip this allocation entirely. We can't assign unknown fuel to a specific bucket.
- If section is **OTHERS** → proceed anyway. OTHERS items like Incinerator don't need a bucket match because they all go into a single `OTHER_Cons` accumulator regardless of fuel type.

**Step D — Accumulate**

Call `result.Add(section, bucketId, amount)` to add the amount to the correct slot in the `FuelAllocationResult` for that `FormId`.

## 10. Step 8 — How FuelAllocationResult Accumulates Totals

**Code:** `DataModels/FuelAllocationResult.cs → Add(section, bucketId, amount)`

`FuelAllocationResult` is the per-form container that holds all classified fuel totals. It has these fields:

| Field | Meaning |
|---|---|
| `ME_HSFO` | Main Engine consumption, HSFO bucket (bucket 1) |
| `ME_VLSFO` | Main Engine consumption, VLSFO bucket (bucket 2) |
| `ME_LSMGO` | Main Engine consumption, LSMGO bucket (bucket 3) |
| `AUX_HSFO` | Auxiliary Engine consumption, HSFO bucket |
| `AUX_VLSFO` | Auxiliary Engine consumption, VLSFO bucket |
| `AUX_LSMGO` | Auxiliary Engine consumption, LSMGO bucket |
| `OTHER_Cons` | All OTHERS section consumption combined (no bucket split) |

The `Add()` method dispatches by (section, bucketId):
- `("ME", 1)` → adds to `ME_HSFO`
- `("ME", 2)` → adds to `ME_VLSFO`
- `("ME", 3)` → adds to `ME_LSMGO`
- `("AUX", 1)` → adds to `AUX_HSFO`
- `("AUX", 2)` → adds to `AUX_VLSFO`
- `("AUX", 3)` → adds to `AUX_LSMGO`
- `("OTHERS", any)` → adds to `OTHER_Cons` (bucket is ignored)

Two computed properties are derived from these fields:
- `ME_Cons_MT` = `ME_HSFO + ME_VLSFO + ME_LSMGO` (total ME regardless of fuel type)
- `AUX_Cons_MT` = `AUX_HSFO + AUX_VLSFO + AUX_LSMGO` (total AUX regardless of fuel type)

These computed totals are what appear in the Excel sheet's ME Total and AUX Total columns.

## 11. Step 9 — Weather Data Stitching

**Code:** `PoolRecapDetailRepository.Mapper.cs → StitchWeatherData()` + `ComputeWeatherValue()`

After fuel classification, the system fetches analyzed weather records and stitches them onto each row.

`GetAnalyzedWeather(formIds)` fetches all `AnalyzedWeather` rows for the given form IDs. Each row is tagged `Is24Hour = true` if it is the single daily summary record, or `false` if it is one of the hourly records.

### Unit Conversion

Before any comparison, raw weather values are converted to the unit system specified in the pool terms:

**Wind:**
- If `WindSpeedUnitId == 2` (Beaufort): run `KnotsToBeaufort(windSpeedUnitId, knots)` — an 11-bracket lookup table converting knot values (0–63+) to Beaufort scale 0–12.
- Otherwise: keep raw knots value.

**Wave:**
- `MetresToDSS(waveHeightUnitId, metres)` — converts wave height to Douglas Sea Scale using the unit system in the pool terms:
  - If `waveHeightUnitId` = metres system: apply DSS thresholds in metres (0, 0.1, 0.5, 1.25, 2.5, 4, 6, 9, 14).
  - If `waveHeightUnitId` = feet system: convert metres to feet first, then apply DSS thresholds in feet.
  - If already in metres (fallback): use metres thresholds.

### ComputeWeatherValue — The Core Algorithm

This function decides what single weather value to assign to a row, and whether that weather was bad.

It is called once for wind and once for wave per row.

It takes:
- `hourlyRows` — all non-24h analyzed weather records for this `FormId`
- `value24h` — the 24h aggregated value for this `FormId`
- `selector` — function to pick either wind or wave from an `AnalyzedWeatherRow`
- `threshold` — the pool term's bad weather threshold (in converted units)
- `badDayHours` — how many bad hours trigger exclusion
- `badDayHoursId` — the type of bad day hours rule
- `steamingHrs` — how long the report period lasted

**Algorithm:**

1. Convert each hourly row's value using the unit conversion functions above.
2. Count how many hourly rows exceed the threshold → `exceedCount`.
3. Apply the **bad day hours** rule:
   - If `badDayHoursId == 1` (fixed hours): is `exceedCount >= badDayHours`? → `hourlyBad = true`.
   - If `badDayHoursId == 2` (ratio): is `exceedCount >= (badDayHours / 24.0) * steamingHrs`? → `hourlyBad = true`.
4. Check the 24h summary value against the threshold → `daily24hBad = (value24h > threshold)`.
5. Decide the final value using this matrix:

| `hourlyBad` | `daily24hBad` | Result |
|:-----------:|:-------------:|--------|
| true | true | Use the 24h summary value directly |
| true | false | Use `threshold + 1` (force it above threshold — the hourly data says it was bad) |
| false | true | Use `threshold` exactly (borderline — don't push above) |
| false | false | Use the 24h summary value directly |

The returned value is written to `row.AnalyzedWind` (Beaufort) or `row.AnalyzedWave` (DSS).

The `Is24Hour` row also supplies direction and current fields which are assigned directly from the 24h record.

## 12. Step 10 — Exemption Calculation (Three-Pass Algorithm)

**Code:** `ExemptionCalculator.cs → Apply(rows)`

The exemption calculator goes through all rows in three passes to assign each row an `ExemptStatus`. This status determines whether a report counts toward the vessel's pool performance.

### Pass 1 — Per-Row Exemption

`ComputeRowExemption(row)` evaluates each non-ATD row in priority order:

1. **`IsPoolExclude` flag set** → `"Excluded"` (manually excluded in the source system)
2. **Wind AND wave both exceeded** → `"Wind & Waves Exceeded"`
3. **Wind only exceeded** → `"Wind Exceeded"`
4. **Wave only exceeded** → `"Waves Exceeded"`
5. **Outside Pool Range** → `"Out side of Pool Range"`
6. **None of the above** → `null` (non-exempt, counts toward pool performance)

**IsWindExceeded logic:**
- Primary check: `row.IsWindExceed == true` (flag from DB / weather aggregation).
- Secondary check: `row.AnalyzedWind > row.WindSpeedThreshold` (both in Beaufort after unit conversion).
- Either condition being true means wind is exceeded.

**IsWaveExceeded logic:**
- Primary check: `row.IsWaveExceed == true`.
- Secondary check: `row.AnalyzedWave > row.WaveHeightThreshold` (both in DSS).

**IsOutsidePoolRange logic:**
- The pool speed for the row's `LoadCondition` (laden or ballast) is compared against the row's `OrderedSpeed`.
- `OrderedSpeed < (poolSpeed - lowerTolerance)` OR `OrderedSpeed > (poolSpeed + upperTolerance)` → excluded.

### Leg Assignment (Between Passes)

Before Pass 2, rows are sorted by `ReportDateTimeUTC`. A `segmentId` counter increments each time an ATD row is encountered. Each row is assigned a `(segmentId, OrderedSpeed)` key. This ensures that short-passage checks don't bleed across separate port-to-port legs.

### Pass 2 — Aggregate Hours Per Leg

Three aggregations are computed per `(segmentId, OrderedSpeed)` key:

- **`gwHours`** — sum of `SteamHours` for rows that are non-ATD and have `ExemptStatus == null` (good weather rows only).
- **`totalHoursPerLeg`** — sum of `SteamHours` for ALL non-ATD rows (including bad weather rows, but not ATD itself).
- **`legsWithBadWeather`** — a HashSet of `(segmentId, OrderedSpeed)` keys where at least one row has a weather-related exemption (Wind/Wave/Wind & Waves).

### Pass 3 — Short Passage Check

For each non-ATD, non-exempt row, check if the passage was too short:

```
if leg has bad weather rows:
    use totalHours (all rows in the leg)
else:
    use gwHours (only good weather rows)

if those hours < MinPassageHours:
    row.ExemptStatus = "Passage too Short"
```

**Why this matters:** The pool terms define a minimum passage duration. If the total time at sea was less than that minimum, all rows in the passage are marked exempt with "Passage too Short", regardless of their individual weather conditions.

The reason we use `totalHours` when bad weather is present — and not just `gwHours` — is that the bad weather rows have already been excluded from pool calculation. If we only counted good-weather hours, a passage with only a few good-weather hours and many bad-weather hours might wrongly be marked as "too short". Using the total passage duration matches the V2 formula which simply checks if total time `H < 24`.

## 13. Step 11 — Passage Grouping and Deduplication

**Code:** `PoolRecapDetailRepository.cs → GetDetailRows()` + `DeduplicatePassages()`

### Grouping Into Passages

`FlushPassage()` is an inner closure that groups rows into ATD-to-ATA blocks. As rows are iterated, a "current passage buffer" accumulates rows until an ATA (Arrival) row is encountered, at which point the buffer is flushed as one passage group.

Passages are further filtered against the requested `PassageIds` — the system calls `GetPassageWindows(passageIds)` which fetches `ActualStartOfSeaPassage` (SOSP) and `ActualEndOfSeaPassage` (EOSP) from the `Voyages` table. Only forms whose `ReportDateTimeUTC` falls within a SOSP–EOSP window are kept.

### Deduplication

`DeduplicatePassages(rows)` groups all rows by a passage key formed from:
- ATD port + ATD datetime
- ATA port + ATA datetime

If two rows point to the same passage key, one is dropped. This handles the case where the same voyage report appears more than once due to duplicate vessel configuration rows in the DB.

### GenerateDocument Pipeline

1. **`WriteDataRows()`** — iterates all passages (newest first), calls `WriteOnePassage()` for each. Returns a `DataWriteResult` containing the last data row number, lists of GW row addresses per load condition, and the pool speeds and multipliers per passage.
2. **`FillHeaderArea()`** — writes vessel name at B2, calls `WriteSummaryFormulaRow()` for the Ballast (row 3) and Laden (row 4) summary rows.
3. **`ClearTemplateShading()`** — clears any template formatting in columns AR+ in header rows.
4. **`ApplyNumericFormat()`** — post-scan every cell in the data area; any cell containing a number or formula gets the configured number format (default `"0.00"`).

### WriteOnePassage

For each passage:
1. Write the ATD row (departure) — location, date, load condition. No fuel or hours.
2. Write each at-sea report (noon reports) — all columns including steam hours, distance, weather, fuel breakdown, exemption status.
3. Write the ATA row (arrival) — location, date. No fuel or hours.
4. Write the **Total row** — formula-based summary for the passage.
5. Write the **GW section** — one or two rows (ballast/laden) with Good Weather performance calculations.

## 15. Formulas Deep Dive

### Total Row Formulas

The total row sits immediately below all data rows for a passage. It summarizes everything using Excel formulas referencing the actual cell range of that passage's rows.

| Column | Formula | What It Calculates |
|--------|---------|-------------------|
| G (Steam Hours) | `=SUM(G{start}:G{end})` | Total steaming hours for the passage |
| H (Distance) | `=SUM(H{start}:H{end})` | Total observed distance (nm) |
| I (SOG) | `=H{r}/G{r}` | Speed over ground = distance ÷ hours |
| J (RPM) | `=SUMPRODUCT(G{start}:G{end}, J{start}:J{end}) / G{r}` | Steam-hours-weighted average RPM |
| K (Slip) | `=SUMPRODUCT(G{start}:G{end}, K{start}:K{end}) / G{r}` | Weighted average slip |
| L (Shaft Power) | `=SUMPRODUCT(G{start}:G{end}, L{start}:L{end}) / G{r}` | Weighted average shaft power |
| M (Current) | `=SUMPRODUCT(G{start}:G{end}, M{start}:M{end}) / G{r}` | Weighted average effective current |
| R–T (ME HSFO/VLSFO/LSMGO) | `=SUM(...)` | Sum of each ME fuel type |
| U (ME Total) | `=SUM(R{r}:T{r})` | Sum of all ME fuel types |
| V–X (AUX HSFO/VLSFO/LSMGO) | `=SUM(...)` | Sum of each AUX fuel type |
| Y (AUX Total) | `=SUM(V{r}:X{r})` | Sum of all AUX fuel types |
| Z (OTHER Cons) | `=SUM(Z{start}:Z{end})` | Total other consumption |
| AA (SOW) | `=SUMPRODUCT(G{start}:G{end}, AA{start}:AA{end}) / G{r}` | Weighted average speed over water |

*`{r}` = the total row number, `{start}:{end}` = the range of data rows in this passage.*

---

### Data Row — Column N (Pool Speed)

Each data row's column N is written as a **static value** (not a formula). It is the pool term speed for the row's load condition:
- If `LoadCondition == "Laden"` → `PoolTermLadenSpeed`
- If `LoadCondition == "Ballast"` → `PoolTermBallastSpeed`

This value is used by the GW row's ME/Day at Pool Speed formula (column W in GW).

---

### GW Row Formulas

The GW (Good Weather) section appears after the Total row for each passage. It may contain up to two rows: one for Ballast, one for Laden (only if the passage had rows in that load condition and the non-exempt hours meet the minimum).

Each GW row uses the passage's `OrderedSpeed` as the SUMIFS key to aggregate only the rows in that passage with that ordered speed and no exemption status.

| Column | Formula | What It Calculates |
|--------|---------|-------------------|
| G (GW Hours) | `=SUMIFS(G_data, C_data, orderedSpd, Q_data, "")` | Steam hours for non-exempt rows at this ordered speed |
| H (GW Distance) | `=SUMIFS(H_data, C_data, orderedSpd, Q_data, "")` | Distance for non-exempt rows |
| I (GW SOG) | `=H{r}/G{r}` | GW speed over ground = GW distance ÷ GW hours |
| J (GW RPM) | `=SUMPRODUCT((C=orderedSpd)*(Q="")*G, J) / G{r}` | Weighted average RPM (good weather rows only) |
| K (GW Slip) | weighted SUMPRODUCT | Weighted average slip |
| L (GW Shaft Power) | weighted SUMPRODUCT | Weighted average shaft power |
| M (GW Current) | weighted SUMPRODUCT | Weighted average current |
| R–T (ME HSFO/VLSFO/LSMGO) | `=SUMIFS(...)` | GW ME fuel totals per fuel type |
| U (ME Total) | `=SUM(R{r}:T{r})` | Total GW ME consumption |
| V–X (AUX) | `=SUMIFS(...)` | GW AUX fuel totals per fuel type |
| Y (AUX Total) | `=SUM(V{r}:X{r})` | Total GW AUX consumption |
| Z (OTHER) | `=SUMIFS(...)` | GW other consumption |
| T (ME/Day at SOG) | `=U{r}/G{r}*24` | ME consumption per day at actual speed |
| W (ME/Day at Pool Speed) | `=(poolSpd/I{r})^multiplier * T{r}` | ME/Day normalized to pool term speed using the power law |

**Column W explained:** The pool evaluates performance at a fixed reference speed (the pool term speed). But the vessel may have sailed at a different ordered speed. The power law `(poolSpd / actualSOG)^multiplier` adjusts consumption to what it would have been at the reference speed. The multiplier (default 2.5) is the propulsion power law exponent from the pool terms.

**`passagePoolSpd`** (the pool speed used in column W) is read from **any row** in the passage (the first non-null match of `PoolTermLadenSpeed` or `PoolTermBallastSpeed` depending on load condition). It does not need to come from a non-exempt row — pool speed is a pool term attribute, not affected by individual row exemptions.

---

### Summary Row Formulas (Rows 3 and 4)

Row 3 = Ballast summary, Row 4 = Laden summary.

These reference the GW row addresses collected during `WriteDataRows()`.

| Column | Formula | What It Calculates |
|--------|---------|-------------------|
| G | `=SUM(GW_G_addresses)` | Total GW hours across all passages for this load condition |
| H | `=SUM(GW_H_addresses)` | Total GW distance |
| I | `=G{r}/H{r}` | Average GW speed = distance ÷ hours |
| J | `=SUM(GW_U_addresses)` | Total ME consumption (all GW passages) |
| K | `=SUM(GW_Y_addresses)` | Total AUX consumption |
| L | `=J{r}/G{r}*24` | ME/Day overall |
| M | `=(poolSpd/I{r})^multiplier * L{r}` | ME/Day at Pool Speed across all passages |
| N | `=K{r}/G{r}*24` | AUX/Day overall |

---

### Data Quality Flags

After writing all rows, `ApplyDataQualityFlags()` scans each data row and marks cells red (`#FF0000` font) in two cases:

- **Column G (Steam Hours) = 0 but fuel is non-zero** → something was consumed but no time was recorded. Likely a data entry error.
- **Column U (ME Total) = 0 or Column Y (AUX Total) = 0, but steam hours > 0** → the vessel was at sea but no fuel was recorded for that engine type.

## 16. All Exemption Labels Explained

| Label | What It Means |
|---|---|
| `"Excluded"` | The report was manually flagged as excluded in the pool system (`IsPoolExclude = true`). |
| `"Wind & Waves Exceeded"` | Both wind (Beaufort) and wave height (DSS) were above the pool term thresholds. |
| `"Wind Exceeded"` | Only wind exceeded the threshold. |
| `"Waves Exceeded"` | Only wave height exceeded the threshold. |
| `"Out side of Pool Range"` | The vessel's ordered speed was outside the allowed tolerance band around the pool term speed. |
| `"Passage too Short"` | The total passage hours were less than the `MinPassageHours` pool term setting. All rows in that passage are marked with this label. |
| `null` / empty | The row is **not exempt**. It counts toward pool performance. These are the "Good Weather" rows. |
